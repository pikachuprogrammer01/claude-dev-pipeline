# Claude Dev Pipeline

A structured development workflow for solo TypeScript frontend projects using
[Claude Code](https://docs.anthropic.com/en/docs/claude-code). Provides a
one-command project initializer that sets up a 9-branch git strategy, branch-
specific CLAUDE files, strict TypeScript enforcement, automated documentation
reminders, and a suite of task-specific skills.

---

## What It Does

Running `@git-init` in Claude Code on a new project:

1. Initializes a git repository with `.claude/project.config.json`
2. Creates 9 branches, each with its own `CLAUDE.md` and `CLAUDE-detailed.md`
3. Installs git hooks for commit validation, doc reminders, and type checking
4. Copies all skills into `.claude/skills/` for project-local use
5. Generates the project `README.md` and syncs it to all branches

---

## Requirements

- macOS, Linux, or Windows
- `git` installed
- `Claude Code` installed
- `PowerShell 5.1+` (Windows only)

## Installation

### macOS / Linux

```bash
curl -fsSL https://raw.githubusercontent.com/pikachuprogrammer01/claude-dev-pipeline/main/install.sh | bash
```

Or from a local clone:

```bash
git clone https://github.com/pikachuprogrammer01/claude-dev-pipeline
cd claude-dev-pipeline
./install.sh
```

### Windows
```bash
irm https://raw.githubusercontent.com/pikachuprogrammer01/claude-dev-pipeline/main/install-Windows.ps1 | iex
```

Or from a local clone:

```bash
git clone https://github.com/pikachuprogrammer01/claude-dev-pipeline
cd claude-dev-pipeline
.\install-Windows.ps1
```

This installs the pipeline to `~/.claude-pipeline/` and registers the
`git-init` skill globally at `~/.claude/skills/git-init/SKILL.md`.

## Starting a New Project

```bash
mkdir my-project && cd my-project
claude                   # open Claude Code
```

In Claude Code:

```
@git-init
```

Claude will ask you to fill in `.claude/project.config.json`, then run the
full setup sequence automatically.

## Branch Strategy

| Branch | Purpose | Skill |
|---|---|---|
| `main` | Production deployment | — |
| `dev` | Feature development | `typescript` |
| `fix` | Bug investigation and fixes | `debug` |
| `refactor` | Code quality improvements | `refactor` |
| `version` | Release management | `release` |
| `test` | Test coverage | `testing` |
| `docs` | Documentation | `docs` |
| `perf` | Performance optimization | `performance` |
| `security` | Security auditing | `security-audit` |

Each branch has its own `CLAUDE.md` (quick reference) and `CLAUDE-detailed.md`
(full methodology). Switching branches gives Claude the right context for that
type of work automatically.

## Document Structure Per Project

```
project/
├── CLAUDE.md              ← branch-specific, auto-written by git-init
├── CLAUDE-detailed.md     ← branch-specific, auto-written by git-init
├── SPEC.md                ← project requirements (shared, fill in manually)
├── README.md              ← generated by docs skill, synced to all branches
├── [branch-specific doc]  ← DEPLOYMENT.md / BUGLOG.md / CHANGELOG.md / etc.
└── .claude/
    ├── project.config.json
    └── skills/
        ├── typescript/SKILL.md
        ├── debug/SKILL.md
        ├── refactor/SKILL.md
        ├── docs/SKILL.md
        ├── release/SKILL.md
        ├── testing/SKILL.md
        ├── performance/SKILL.md
        └── security-audit/SKILL.md
```

## Git Hooks

Three hooks are installed automatically:

| Hook | When | What It Does |
|---|---|---|
| `commit-msg` | Every commit | Enforces Conventional Commits format |
| `post-commit` | After each commit | Prints documentation update reminders based on changed files |
| `pre-push` | Before push | Runs `tsc --noEmit` and verifies branch-required documents exist |

To bypass in an emergency: `git push --no-verify`

---

## TypeScript Enforcement

The `typescript` skill and `dev` branch `CLAUDE.md` enforce a strict type
hierarchy. From highest to lowest preference:

1. **Branded types** — domain concepts (`UserId`, `Price`, `Timestamp`)
2. **Literal unions** — fixed option sets (`'active' | 'inactive'`)
3. **Discriminated unions** — state machines, result types, async states
4. **Template literal types** — constrained string shapes (`/users/${string}`)
5. **Generics with constraints** — reusable utilities
6. **Primitives** — last resort only

`any`, `as` casts in application logic, and unvalidated external data are not
permitted. All external data enters through Zod schemas.

Active compiler options beyond `strict`:
- `noUncheckedIndexedAccess`
- `exactOptionalPropertyTypes`
- `noPropertyAccessFromIndexSignature`

## Customization

**Project-specific settings** — edit `.claude/project.config.json` in each project:

```json
{
  "projectName": "my-project",
  "framework": "next",
  "packageManager": "pnpm",
  "testRunner": "vitest",
  "srcDir": "src",
  "deployTarget": "vercel",
  "cssApproach": "tailwind"
}
```

**Skill content** — edit files in `~/.claude-pipeline/skills/`. Run
`update.sh` to pull upstream changes (your local edits will need to be
managed as a fork or separate branch).

**CLAUDE.md templates** — edit files in `~/.claude-pipeline/templates/claude/`
and `templates/claude-detailed/`.

## Updating

### macOS / Linux
```bash
~/.claude-pipeline/update.sh
```

Pulls the latest version and refreshes the global `git-init` skill.
Existing projects keep their currently installed skill versions.

To refresh skills in an existing project:

```
@git-init --refresh
```

### Windows
```bash
~\.claude-pipeline\update-Windows.ps1
```

---

## Uninstalling

### macOS / Linux
```bash
~/.claude-pipeline/uninstall.sh
```

Removes `~/.claude-pipeline/`, the global `git-init` skill, and the path
marker. Existing project `.claude/` directories are not affected.

### Windows

```bash
~\.claude-pipeline\uninstall-Windows.ps1
```
